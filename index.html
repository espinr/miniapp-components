MiniApp UI components are the building blocks to create MiniApp pages. Each component encapsulates functionality, data and styles, enabling developers to create custom and reusable items that can be combined to create MiniApps. MiniApps include essential UI components like page containers, textual and multimedia content and other interactive elements such as forms.

This document specifies a set of common UI components for MiniApps that allows developers to build similar user interfaces across MiniApp platforms efficiently.

Introduction

MiniApp UI components

MiniApp is a concept of light software application that can be distributed through any digital means and accessed through the Web. MiniApps are composed of one or more pages, and the pages are defined using high-level building blocks called MiniApp UI Components.

A MiniApp UI component is the minimum building block to construct the User Interface and functionality of a MiniApp page.

As specified in [[MINIAPP-PACKAGING]], a MiniApp page is composed of, at least, an HTML resource and, optionally, a CSS resource and/or a scripting resource. Developers use these resources to declare the essential UI components, define the visual appearance and user interaction functions, and program their behavior.

Summary of components

MiniApp UI components can be classified into basic components, essential blocks to deliver audiovisual content to the user, and containers for structuring the content.

Attributes

As in HTML elements, MiniApp Components may have content attributes [[HTML]]. Except where otherwise specified, attribute values may be any string value, including the empty string.

Global attributes

The following attributes are common to all MiniApp components:

  • [=component/`id`=]
  • [=component/`class`=]
  • [=component/`style`=]
<div class="content">
    <text id="title" style="font-size: 15px; color: blue;"> 
        Hello World
    </text>
</div>

The `id` attribute

The `id` attribute specifies the unique identifier of the component in the document. The [=component/id=] attribute value MUST contain at least one character. The value MUST not contain any ASCII whitespace.

The `class` attribute

The `class` attribute specifies the classes of a component, affecting matching in selectors in CSS.

As specified in the HTML elements [[HTML]], the [=component/`class`=] attribute MUST have a value that is a set of space-separated tokens representing the various classes that the element belongs to.

The `style` attribute

The `style` attribute specifies the style of a [=component=].

Events

Events are dispatched by the user agent as the result of user interaction, change of status in the application, or the completion of tasks.

Common Event Types

All [=MiniApp UI components=] implement the [=component/`BasicEvent`=] target interface that allows registration and removal of event listeners and handling functions on specific events.

This section specifies the types of events supported by all [=MiniApp UI components=] as event targets.

Events that can be triggered on any component
Event Type Interface Description
[=component/`click`=] [=component/`BasicEvent`=] Event dispatched when the user presses down and releases the primary pointer.
[=component/`longpress`=] [=component/`BasicEvent`=] Event dispatched when the user presses down and holds the primary pointer for a specific period of time (@@@ms?).
[=component/`swipe`=] [=component/`BasicEvent`=] Event dispatched when the user presses down the primary pointer and slides it without releasing it.
[=component/`touchstart`=] [=component/`TouchEvent`=] Event dispatched when the user starts pressing down the primary pointer.
[=component/`touchmove`=] [=component/`TouchEvent`=] Event dispatched when the user keeps pressing down and moves the primary pointer.
[=component/`touchcancel`=] [=component/`TouchEvent`=] Event dispatched when the touch operation was interrupted by the system.
[=component/`touchend`=] [=component/`TouchEvent`=] Event dispatched when the user releases the primary pointer.

`BasicEvent` Interface

The following event types implement the [=component/`BasicEvent`=] interface.

The `click` event

The `click` event type MUST be dispatched on the topmost component (event target) indicated by the pointer when the user presses down and releases the primary pointer button or otherwise activates the pointer in a manner that simulates such an action.

See Mouse Event Types (click).

The `longpress` event

The `longpress` event type MUST be dispatched on the topmost component (event target) indicated by the pointer, when the user presses down, holds for at least (@@@ms@@@) and releases the primary pointer button, or otherwise activates the pointer in a manner that simulates such an action.

To define the minimum period to be considered as a long press.

The `swipe` event

The `swipe` event type MUST be dispatched on the topmost component (event target) indicated by the pointer when the user presses down, holds, and moves across the component using the primary pointer button, or otherwise activates the pointer in a manner that simulates such an action.

Apart from the attributes of the [=component/`BasicEvent`=] interface, the [=component/`swipe`=] event implements the attribute `direction`. The `direction` attribute is a string that indicates the direction of the swiping movement of the user, with the following possible values:

`left`
Horizontal movement from right to left.
`right`
Horizontal movement from left to right.
`up`
Vertical ascendant movement.
`left`
Vertical descendant movement.

`TouchEvent` Interface

The `TouchEvent` interface defines the `touchstart`, `touchend`, `touchmove`, and `touchcancel` event types.

IDL definition of the interface? See TouchEvent Interface.

Apart from the attributes of the [=component/`BasicEvent`=] interface, the [=component/`TouchEvent`=] interface contains the following attributes:

`touches`
A list of `Touch` objects for every point of contact currently touching the surface.
`changedTouches`
A list of `Touch` objects for every point of contact which contributed to the event.

The following event types implement the `TouchEvent` interface.

The `touchstart` event

The `touchstart` event type MUST be dispatched on the topmost component (event target) indicated by the pointer when the user places a touch point on the touch surface.

The `touchmove` event

The `touchmove` event type MUST be dispatched on the topmost component (event target) indicated by the pointer when the user moves a touch point along the touch surface.

The `touchcancel` event

The `touchcancel` event type MUST be dispatched on the topmost component (event target) indicated by the pointer when the touch point has been disrupted in a implementation-specific manner.

The `touchend` event

The `touchend` event type MUST be dispatched on the topmost component (event target) indicated by the pointer when the user removes a touch point from the touch surface, also including cases where the touch point physically leaves the touch surface.

`Touch` Interface

interface Touch {
    readonly        attribute double      globalX;
    readonly        attribute double      globalY;
    readonly        attribute double      localX;
    readonly        attribute double      localY;
    readonly        attribute double      size;
};

The `Touch` interface contains the following attributes:

`globalX`
The horizontal coordinate of point relative to the screen in pixels (excluding the status bar, if present).
`globalY`
The vertical coordinate of point relative to the screen in pixels (excluding the status bar, if present).
`localX`
The horizontal coordinate of point relative to the target component in pixels.
`localY`
The vertical coordinate of point relative to the target component in pixels.
`size`
The touch area (@@@ units).
Similar to Touch.screenX, Touch.screenY, ...

`refresh`

The `refresh` component is a container to implement the pull-to-refresh mechanism on a page or list of items.

In addition to the global attributes, the [=component/`refresh`=] component defines the following content attributes:

`refresh` component additional attributes
Attribute Type Value by default Mandatory Description
offset `number` no Distance to the top of the parent component from the `refresh` component that specifies a successful swipe gesture.
type `string` `auto` no Dynamic effect when the component is refreshed. Two options are available and cannot be modified dynamically:
  • `auto` (default behavior): when the container is pulled to the top, there is no effect; when the container is pulled down, the refresh is activated, and a loading indicator is shown
  • `pulldown`: when the list is pulled to the top, users can continue to pull down to trigger a refresh.
refreshing `boolean` `false` no Flag that indicates if the component is being refreshed or not.

friction `number` no Pull-down friction coefficient. The value ranges from `0` to `100` (larger values indicate more responsiveness).

disabled `boolean` `false` no Whether the component is disabled. If it is disabled, it cannot respond to user interaction.
focusable `boolean` `true` no Attribute to specify if the component is a focusable area (value `true`) or not (value `false`).
data `string` no Auxiliary attribute to facilitate data storage and manipulation.
The rebound effect will appear after the refresh. If the child component contains a list, set `scrolleffect` of the list to `false` to prevent drop-down effect conflicts.

`tab-bar`

The `tab-bar` component is a child of `tabs` and provides the area to display tab labels.

`tab-bar` implements the same attributes as `tabs`. In addition to these, `tab-bar` components define the following attributes:

`tab-bar` component additional attributes
Attribute Type Value by default Mandatory Description
mode `string` `scrollable` no Extensibility of the component width. Available values are as follows:
  • `scrollable`: The width of a child component is the configured width. When the total width of all child components is greater than the tab-bar width, the child component can scroll horizontally.
  • `fixed`: The width of a child component equals the tab-bar width divided by the number of child components.

`tab-content`

The `tab-content` is the child of `tabs` that provides the area for displaying the tab content. By default, its height is such that all the remaining space of the `tabs` component is filled. The child components are arranged horizontally. When `tab-content` is used as a child element in a container, the length on the main axis direction MUST be specified. Otherwise, the child element cannot be displayed.

`tab-content` implements the same attributes as `tabs`. In addition to these, `tab-content` components define the following attributes:

`tab-content` component additional attributes
Attribute Type Value by default Mandatory Description
scrollable `boolean` `true` no Flag that indicates if the tabs can be switched by swiping on the tab content. The default value is `true`. If this attribute is set to `false`, tab switching is implemented only through the interaction with `tab-bar`.
